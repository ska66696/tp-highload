## 1. Тема и целевая аудитория
**Discord** - платформа для текстового и голосового общения в рамках пользовательских сообществ (серверов).

### Целевая аудитория
*   **MAU (Monthly Active Users):** 200 млн [^1].
*   **DAU (Daily Active Users):** 26.5 млн [^3].
*   **Количество зарегистрированных аккаунтов:** 560 млн [^2].
*   **География:** Глобальный рынок. Распределение трафика по данным на начало 2026 года [^4]:

| Страна | Доля трафика (%) | Оценка MAU (млн чел) |
| :--- | :--- | :--- |
| США | 29.1% | 58.2 |
| Бразилия | 7.25% | 14.5 |
| Индия | 4.9% | 9.8 |
| Германия | 3.2% | 6.4 |
| Канада | 2.7% | 5.4 |

![Трафик по странам](images/traffic.png) 
*Распределение трафика по странам (Semrush) [^4]*

### Функционал MVP
Основной функционал заключается в создании иерархических сообществ и обеспечении каналов связи внутри них.

1.  **Управление серверами:** создание серверов и текстовых/голосовых каналов.
2.  **Текстовые чаты:** обмен сообщениями в реальном времени внутри каналов.
3.  **Голосовые каналы:** аудио-сессии для группового общения.
4.  **История сообщений:** хранение и доступ к переписке на стороне сервера.
5.  **Статусы:** статусы пользователей («в сети», «играет в...»).
6.  **Вложения (Media):** возможность загрузки изображений и файлов в текстовые каналы.

### Ключевые продуктовые решения
*   Голосовые каналы работают по модели «постоянной комнаты».
*   История сообщений хранится на серверах бессрочно и доступна с любого устройства в любой момент.
*   Синхронизация состояния серверов, сообщений и статусов между всеми активными сессиями пользователя.

## 2. Расчет нагрузки

### Продуктовые метрики
Базовые показатели аудитории, на которые мы опираемся в расчетах:
*   **MAU (Monthly Active Users):** 200 млн [^1]
*   **DAU (Daily Active Users):** 26.5 млн [^3]
*   **Зарегистрированных аккаунтов:** 560 млн [^2]
*   **Активных серверов (Guilds):** 21 млн [^2]

**Средний размер хранилища пользователя**
Discord хранит более 1.5 триллиона сообщений [^5]. Среднее количество на аккаунт: `1.5 трлн / 560 млн ≈ 2678`.
*   *Текст:* Средний размер сообщения составляет ~250 байт [^10].
*   *Вложения:* Опираясь на статистику мессенджеров (WhatsApp [^12]), доля сообщений с медиа составляет ~12%. Для игрового чата примем оценку **10%**. Лимит загрузки для бесплатных пользователей - 10 МБ [^8], средний размер файла примем за 1 МБ.
*   *Профиль:* Аватар (~100 КБ) + настройки.

| Тип данных | Объем (шт) | Объем на 1 пользователя |
| :--- | :--- | :--- |
| Текстовые сообщения (история) | 2 678 | ~670 КБ (`2678 * 250 байт`) |
| Вложения (картинки, файлы) | 268 | ~268 МБ (`10% от сообщений * 1 МБ`) |
| Профиль | 1 | ~100 КБ |
| **Итого на пользователя** | | **~269 МБ** |

**Среднее количество действий пользователя в день**
1.  **Управление серверами**
    *   Лимит для бесплатного аккаунта - 100 серверов [^13]. Пользователи редко вступают в новые сервера или создают их. Примем частоту действия за 1 раз в неделю (0.14 в день).
2.  **Текстовые чаты (Отправка)**
    *   В день через платформу проходит **4 млрд сообщений** [^11]. В пересчете на активного пользователя: `4000 млн / 26.5 млн ≈ 150`.
    *   *Обоснование:* Цифра включает в себя сообщения от ботов и системные уведомления, которые инфраструктура обязана обработать и сохранить.
3.  **Голосовые каналы**
    *   Трафик составляет ~8 млрд минут в день (экстраполяция данных 2020 года [^6] на текущий MAU). На одного пользователя: `8000 млн / 26.5 млн ≈ 301 минута` (около 5 часов).
4.  **История сообщений (Чтение)**
    *   Пользователи открывают приложения мессенджеров в среднем 9 раз в день [^14].
    *   Согласно документации API Discord, при открытии канала клиент запрашивает пакет из **50 сообщений** по умолчанию [^15].
    *   Итого чтений (Loading): `9 сессий * 50 сообщений = 450 сообщений/день`.
5.  **Статусы и Fan-out**
    *   **Обоснование частоты (10 раз):** Складывается из сценария использования:
        1.  Вход/выход из приложения (2 события).
        2.  Игровая активность: согласно статистике [^16], средний геймер совершает **2.4 игровые сессии в день**. Каждая сессия генерирует статус "Начал играть" и "Закончил играть" (`2.4 * 2 ≈ 5` событий).
        3.  Переходы в режим "Не активен" (AFK) при перерывах (~3 события).
        4.  *Итог:* `2 + 5 + 3 = 10`.
    *   Учитывая архитектуру пассивных сессий [^9], сообщение доставляется не всем участникам сервера, а только активным. Примем эффективный fan-out = 20.
6.  **Вложения (Media)**
    *   Исходя из доли медиа-сообщений 10% [^12], пользователь загружает `150 * 0.1 = 15` файлов в день.

| Действие (MVP) | Количество в день | Обоснование |
| :--- | :--- | :--- |
| Управление серверами | 0.14 | 1 раз в неделю (лимит 100 серверов [^13]) |
| Отправка сообщений | 150 | Общая статистика (вкл. ботов) [^11] |
| Голосовая связь (минут) | 301 | Экстраполяция оф. данных [^6] |
| Чтение истории (сообщений) | 450 | 9 сессий [^14] * 50 msg limit [^15] |
| Смена статуса | 10 | Смена деятельности в течение дня |
| Загрузка вложений | 15 | 10% от отправленных сообщений [^12] |

### Технические метрики

**Обоснование пикового коэффициента (k = 1.5)**
Сервис является глобальным, поэтому нагрузка распределена по часовым поясам. Однако, согласно статистике трафика [^4], **~40%** аудитории сосредоточено в Северной и Южной Америке (США, Бразилия, Канада). Вечерний прайм-тайм этого макрорегиона создает основную волну нагрузки.
*   *Решение:* Используем коэффициент пиковой нагрузки **1.5**, так как остальные 60% трафика размазаны по Европе и Азии.

#### Размер хранилища
Discord хранит историю сообщений вечно. Расчет строится на накопленных данных (1.5 триллиона сообщений [^5]).

*   **Текстовые сообщения:** 1.5 трлн записей. Средний размер сжатого пейлоада - **250 байт** (по графику из инженерного блога [^10]).
*   **Медиа-вложения:** Согласно глобальной статистике мессенджеров (WhatsApp [^12]), доля сообщений с медиа составляет ~12%. Для игрового чата примем оценку **10%**.
    *   *Количество:* `1.5 трлн * 0.10 = 150 млрд файлов`.
    *   *Размер:* Лимит для бесплатных пользователей - 10 МБ [^8]. Средний размер файла (скриншоты, мемы) оценим в **1 МБ**.
*   **Метаданные серверов (Guilds):** 21 млн активных серверов [^2]. Храним роли, каналы, настройки. Средний размер объекта - 500 КБ.

| Тип данных | Количество (штук) | Расчет объема | Итоговый размер |
| :--- | :--- | :--- | :--- |
| **Текстовые сообщения** | 1.5 трлн | `1.5 * 10^11 * 250 байт` | **375 ТБ** |
| **Медиа-вложения** | 150 млрд | `150 * 10^9 * 1 МБ` | **150 ПБ** |
| **Профили пользователей** | 560 млн | `560 * 10^6 * 100 КБ` | **56 ТБ** |
| **Метаданные серверов** | 21 млн | `21 * 10^6 * 500 КБ` | **10.5 ТБ** |
| **Итого** | | | **~150.4 ПБ** |

#### Сетевой трафик
Расчет суточного и пикового потребления каналов связи.

*   **Голосовой трафик (Voice):**
    *   *Обоснование:* В 2020 году (100 млн MAU) трафик составлял 4 млрд минут/день [^6]. Экстраполируя на 200 млн MAU, получаем **8 млрд минут в день**.
    *   *Битрейт:* Стандартное качество - **64 kbps** (8 КБ/с) [^7].
*   **Медиа-трафик (CDN):**
    *   *Обоснование:* В сутки отправляется 4 млрд сообщений [^11]. 10% из них содержат файлы (400 млн файлов).
    *   *Объем:* `400 млн * 1 МБ = 400 ТБ` (загрузка). С учетом скачивания другими пользователями (Read factor ~3), суточный обмен ~1.2 ПБ.
*   **Текстовый Fan-out (WebSocket):**
    *   *Обоснование:* Сообщения доставляются через WebSocket. Благодаря архитектуре "пассивных сессий" [^9], сообщение не рассылается всем 1000 участникам сервера, а только тем, кто активен/смотрит экран. Примем эффективный **Fan-out = 20** (1 сообщение получают 20 человек).

| Тип трафика | Суммарный суточный (ГБ/сутки) | Средняя полоса (Гбит/с) | Пиковая полоса (Гбит/с) k=1.5 |
| :--- | :--- | :--- | :--- |
| **Голос (UDP)** | 3 840 000 | 355.5 | 533.3 |
| **Медиа (HTTP)** | 1 200 000 | 111.1 | 166.7 |
| **Fan-out (WS)** | 20 000 | 1.9 | 2.9 |
| **Итого** | **~5.06 ПБ/сут** | **~468.5 Гбит/с** | **~702.9 Гбит/с** |

#### RPS (Запросы в секунду)
Расчет нагрузки на бэкенд по типам запросов.
*   **Входящие сообщения (Write):** 4 млрд сообщений в сутки [^11].
    *   *Обоснование:* Эта цифра включает сообщения от ботов и системные уведомления.
*   **Чтение истории (Load History):**
    *   *Обоснование:* В среднем пользователь открывает приложение/меняет чат **9 раз в день** [^14].
    *   *Лимит:* По умолчанию API Discord запрашивает пакет из **50 сообщений** (`limit=50`) [^15].
    *   *Расчет:* `26.5 млн DAU * 9 сессий = 238.5 млн запросов`.
*   **Доставка (Push):** Работает через WebSocket push.
    *   *Расчет:* `Write RPS * Fan-out (20)`.

| Тип запроса | Обоснование расчета (сутки / 86400) | Средний RPS | Пиковый RPS (x1.5) |
| :--- | :--- | :--- | :--- |
| **Отправка сообщения** (Write) | `4 млрд / 86400` | 46 296 | 69 444 |
| **Доставка / Fan-out** (Push) | `Write RPS * 20` | 925 920 | 1 388 880 |
| **Загрузка истории** (Read API) | `(26.5 млн * 9) / 86400` | 2 760 | 4 140 |
| **Загрузка файлов** (Upload) | `10% от Write RPS` | 4 630 | 6 945 |
| **Обновление статуса** (Presence) | `(26.5 млн * 10) / 86400` | 3 067 | 4 600 |
| **Управление серверами** | `(26.5 млн * 0.14) / 86400` | 43 | 65 |

## 3. Глобальная балансировка нагрузки

### Функциональное разбиение по доменам
В целях изоляции различных типов трафика инфраструктура разбита на следующие домены:
*   `discord.com/api` - для обработки стандартных HTTP/REST запросов (авторизация, управление настройками профиля и серверов).
*   `gateway.discord.gg` - для поддержания постоянных WebSocket-соединений (Gateway). Обеспечивает передачу текстовых сообщений, статусов и сигналинга звонков в реальном времени.
*   `cdn.discordapp.com` - для первоначальной загрузки и хранения оригинальных пользовательских статических файлов и медиа-вложений.
*   `media.discordapp.net` - для отдачи оптимизированных медиа-файлов (ресайз изображений, генерация превью для видео). Выделение отдельного домена позволяет эффективно кэшировать производные форматы и экономить пропускную способность.
*   `*.discord.media` (например, `frankfurt.discord.media`) - домены голосовых серверов для маршрутизации UDP-трафика реального времени (WebRTC).

### Расположение ДЦ
Голосовая связь требует задержки (RTT) не более 50 мс. Поэтому инфраструктура размещена в 13 дата-центрах, выбранных с учетом пересечения подводных интернет-магистралей[^20] и доступа к надежным источникам электроэнергии [^21]. 

1.  **Северная Америка (4 ДЦ):** Ашберн (Восток), Чикаго (Центр), Лос-Анджелес (Запад), Майами (Юг). Выбраны из-за высокой доли аудитории (29.1%) и наличия главных магистральных узлов.
2.  **Европа (3 ДЦ):** Франкфурт, Лондон, Амстердам. Крупнейшие европейские точки обмена трафиком.
3.  **Азия (3 ДЦ):** Мумбаи, Сингапур, Токио. Основные кабельные хабы Азии и Юго-Восточной Азии.
4.  **Южная Америка (1 ДЦ):** Сан-Паулу. Обслуживает 7.25% глобальной аудитории (Бразилия) и является основным узлом высадки кабелей континента.
5.  **Океания (1 ДЦ):** Сидней. Необходим из-за географической изоляции региона.
6.  **Африка (1 ДЦ):** Йоханнесбург. Обусловлен необходимостью локального транзита медиа-трафика для Африки.

### Расчет распределения запросов по ДЦ
Каждый дата-центр рассчитан на обработку $1 / 13$ от глобального пикового трафика (общий пиковый RPS составляет 1 474 074). В регионах с повышенной плотностью пользователей запросы распределяются между несколькими локальными ДЦ, что выравнивает нагрузку.

| Регион | Количество ДЦ | Суммарный Пик RPS на регион | Пик RPS на 1 ДЦ | Суммарный Пик Трафика на регион (Гбит/с) | Пик Трафика на 1 ДЦ (Гбит/с) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| Северная Америка | 4 | 453 560 | ~113 390 | 216.3 | ~54.07 |
| Европа | 3 | 340 170 | ~113 390 | 162.2 | ~54.07 |
| Азия | 3 | 340 170 | ~113 390 | 162.2 | ~54.07 |
| Южная Америка | 1 | 113 390 | 113 390 | 54.07 | 54.07 |
| Океания | 1 | 113 390 | 113 390 | 54.07 | 54.07 |
| Африка | 1 | 113 394 | 113 394 | 54.07 | 54.07 |
| **Итого** | **13** | **1 474 074** | | **702.9** | |

### Схема DNS балансировки
Используется Geo-based / Latency-based DNS для первичного направления пользователя в географически ближайший доступный дата-центр с минимальным RTT.

### Схема Anycast балансировки
Домены `api`, `gateway`, `cdn` и `media` работают поверх сети Cloudflare с использованием маршрутизации BGP Anycast[^19]. Один IP-адрес анонсируется из всех ДЦ, и запрос клиента маршрутизируется протоколом BGP на топологически ближайший сервер.

### Механизм регулировки трафика между ДЦ
Для передачи голосового UDP-трафика (WebRTC) Anycast не применяется, так как перестроение BGP-маршрутов может привести к обрыву потоковых сессий. Вместо этого используется Unicast-маршрутизация к конкретным серверам `*.discord.media`.

Механизм регулировки и перераспределения голосового трафика между ДЦ реализован через управляющий микросервис (`Discord Guilds`) и Service Discovery (`etcd`). Сервис непрерывно агрегирует метрики загруженности всех 13 дата-центров. В случае исчерпания ресурсов конкретного дата-центра или DDoS-атаки на него, управляющий сервис автоматически выбирает наименее загруженный ДЦ в регионе и отправляет клиентам команду (через Gateway) на установку нового UDP-соединения[^18].

## 4. Локальная балансировка нагрузки

### Схема балансировки нагрузки
Внутри каждого дата-центра маршрутизация запросов разделена на обработку внешнего (входящего) и внутреннего (межсервисного) трафика. 

**Уровни внешней балансировки:**
1.  **L4-балансировка (Транспортный уровень):** 
    В качестве единой точки входа используется Software Load Balancer **LVS (Linux Virtual Server)** в режиме **Virtual Server via Direct Routing (DR)**. 
    *   *Обоснование режима DR:* Весь обратный трафик (особенно тяжелый исходящий голосовой UDP-поток) идет от серверов к клиенту напрямую, минуя балансировщик. Это предотвращает перегрузку сети на самом балансировщике.
    *   *Маршрутизация:* Голосовой трафик (UDP) L4-балансировщик отправляет напрямую на bare-metal голосовые серверы (минуя L7). HTTP и WebSocket трафик (TCP) отправляется на слой L7-балансировщиков.
2.  **L7-балансировка (Прикладной уровень):**
    Используются серверы **NGINX**. Они осуществляют TCP multiplexing, поддерживают постоянные WebSocket-соединения и выполняют SSL Termination (расшифровку HTTPS). 
    *   *Оптимизация:* Для снижения нагрузки на CPU при SSL-хэндшейках применяются **Session tickets**. Расшифрованный трафик маршрутизируется на внутренние микросервисы.

**Механизмы резервирования:**
*   **L4 (LVS):** Используется схема резервирования **N * 2 (Active-Passive)**. Для мониторинга доступности нод и автоматического переключения IP-адресов в случае сбоя применяется демон `keepalived` (с использованием протокола VRRP).
*   **L7 (NGINX):** Используется схема резервирования **N + 1 (Active-Active)**. Все балансировщики работают одновременно, при падении одного из них трафик равномерно распределяется между оставшимися.

**Межсервисная (внутренняя) балансировка:**
Для маршрутизации внутреннего трафика между микросервисами применяется паттерн **L7 локальный на клиенте (Sidecar proxy)** на базе Envoy. Каждый микросервис имеет свой локальный прокси, который обеспечивает клиентскую балансировку, service discovery, ретраи и настройку таймаутов.

### Расчет количества балансировщиков
Расчет производится для одного дата-центра на основе пиковых показателей из Раздела 3:
*   Суммарный трафик на 1 ДЦ: **~54.06 Гбит/с**
*   Трафик L7 (HTTP + WS, без голоса): **~13.04 Гбит/с**
*   Пиковый RPS L7 (HTTP + WS): **~113 390 RPS**

**Эталонные показатели сервера (согласно бенчмаркам NGINX):**
Сервер на базе 16 CPU с включенным Hyper-Threading и сетевой картой 10GbE обеспечивает:
*   Пропускная способность (HTTP Throughput): **8.8 Гбит/с**
*   Терминация SSL (SSL TPS для RSA 2K): **56 715 соединений в секунду**

**1. Расчет балансировщиков L7 (NGINX):**
Ограничителями для L7 выступают терминация SSL и пропускная способность сети.
*   *По ограничению SSL:* $113 390 \text{ RPS} / 56 715 \text{ TPS} = 1.99 \rightarrow \text{2 сервера}$. (Расчет взят с запасом, так как благодаря Session tickets и постоянным WebSocket-соединениям реальное количество *новых* TLS-хэндшейков ниже общего RPS).
*   *По пропускной способности:* $13.04 \text{ Гбит/с} / 8.8 \text{ Гбит/с} = 1.48 \rightarrow \text{2 сервера}$.
*   *Итог для L7:* По обоим ограничителям требуется 2 активных сервера. Применяем формулу резервирования **N + 1**: $2 + 1 = \mathbf{3}$ **сервера NGINX на один ДЦ**.

**2. Расчет балансировщиков L4 (LVS):**
Ограничителем для L4 выступает только пропускная способность сетевых интерфейсов, так как SSL-терминация не выполняется. Для L4 серверов используются сетевые карты 40GbE.
*   *По пропускной способности:* $54.06 \text{ Гбит/с} / 40 \text{ Гбит/с} = 1.35 \rightarrow \text{2 сервера}$.
*   *Итог для L4:* Требуется 2 активных сервера. Применяем формулу резервирования **N * 2**: $2 \times 2 = \mathbf{4}$ **сервера LVS на один ДЦ**.

### Требования к консистентности

**Строгая консистентность (Strict Consistency):**
Ограничивается исключительно критическими узлами, связанными с безопасностью аккаунтов, чтобы избежать дорогостоящих распределенных блокировок:
*   `users` - смена пароля или привязка почты должна применяться немедленно.
*   `sessions` - при отзыве токена (нажатие "Выйти со всех устройств") доступ должен прекращаться мгновенно во всех дата-центрах для защиты от компрометации.

**Согласованность в конечном счете (Eventual Consistency):**
Применяется для абсолютного большинства данных системы. В условиях глобально распределенной архитектуры задержка репликации в 1-2 секунды является допустимой платой за высокую доступность и низкий latency (согласно теореме CAP):
*   `messages` и `attachments` - небольшая задержка при доставке сообщения другим участникам.
*   `presences` - статусы друзей могут обновляться асинхронно.
*   `guild_members` и `roles` - если пользователю выдали новую роль или он вступил на сервер, задержка применения этих прав в 1-2 секунды не влияет на UX, но позволяет снять оверхед с базы данных.
*   `relationships` - принятие заявки в друзья или блокировка пользователя также могут реплицироваться в фоне.

### Особенности распределения нагрузки (Шардирование)
1.  **messages:** Основной ключ шардирования - `channel_id`. Это обеспечивает высокую локальность данных: сообщения одного канала физически находятся в одном шарде, что позволяет эффективно выбирать историю последних 50 сообщений без сканирования всей базы.
2.  **users и presences:** Шардируются по `user_id`. Это равномерно распределяет запросы на авторизацию и обновление статусов по узлам кластера.
3.  **guild_members:** Шардирование по `guild_id`. Необходимо для быстрой рассылки уведомлений всем участникам конкретного сервера при публикации сообщения.
4.  **sessions:** Шардирование по `token` (или его хэшу). Это обеспечивает максимально быстрое чтение и проверку прав доступа (auth middleware) при каждом входящем HTTP/WebSocket запросе, равномерно размазывая нагрузку (120 000 QPS) по кластеру in-memory БД.

### Шардирование и Партиционирование

Выбор ключей шардирования обусловлен необходимостью обеспечения локальности данных для предотвращения scatter-gather запросов (веерной рассылки запроса по всем узлам кластера).

1. **Шардирование по функциональному признаку (Entity-based):**
   * **Ключ `channel_id` (таблица `messages`):** Основной паттерн чтения - выборка истории конкретного канала. Использование `channel_id` как Shard Key направляет запрос строго в один физический шард кластера, предотвращая веерную рассылку (scatter-gather query).
   * **Ключ `guild_id` (таблица `guild_members`):** Все 1.5 ТБ данных об участниках распределяются так, чтобы список участников конкретного сервера физически хранился на одном узле.

2. **Шардирование по идентификатору (Hash-based):**
   * **Ключ `user_id` (таблица `relationships`):** Хэширование ключа обеспечивает равномерное распределение 22.4 млрд записей по кластеру. Благодаря двунаправленной записи связей, запрос списка друзей пользователя всегда попадает только на один шард.

3. **Партиционирование (Partitioning):**
   * Применяется для таблицы `messages` внутри каждого шарда. Используется Range Partitioning по полю `id` (Snowflake ID), которое содержит временную метку (timestamp). Позволяет эффективно управлять жизненным циклом данных и архивировать старые партиции.

### Резервирование

Для каждой СУБД выбрана схема горячего резервирования для исключения единой точки отказа:

* **PostgreSQL:** Схема **Master-Slave** с асинхронной репликацией. Запись осуществляется в Master, чтение толерантных к задержке данных - со Slave-узлов.
* **MongoDB:** Использование **Replica Set** (1 Primary + 2 Secondary). Обеспечивает автоматическое переключение (failover) при отказе узла. Реплики распределяются между разными стойками в дата-центре.
* **Redis:** Схема **Master-Slave + Sentinel**. Sentinel отслеживает состояние узлов и автоматически назначает нового мастера при сбое.

### Клиентские библиотеки и интеграции

Для работы с распределенными СУБД на уровне приложения используются драйверы с поддержкой пулов соединений:

* **PostgreSQL:** Библиотека `pgx` (Go). Поддерживает работу с массивами и автоматическую подготовку стейтментов.
* **MongoDB:** Официальный `mongo-go-driver`. Используется для асинхронной записи сообщений.
* **Redis:** Библиотека `go-redis`. Поддерживает работу с Redis Sentinel для автоматического переключения на Slave-узлы при сбоях.
* **Kafka (Шина событий):** Библиотека `confluent-kafka-go`. Используется как промежуточный буфер для входящего потока сообщений и передачи событий Presence (статусов).

### Балансировка запросов и мультиплексирование

1. **PostgreSQL (pgBouncer):** Перед каждым шардом Postgres установлен **pgBouncer**. Выполняет мультиплексирование соединений: тысячи TCP-соединений от бэкенд-сервисов транслируются в ограниченный пул соединений к СУБД, снижая накладные расходы на создание процессов в PostgreSQL.
2. **MongoDB (mongos):** Балансировка и маршрутизация осуществляются через компонент **mongos**. Процесс кэширует карту шардов и перенаправляет запрос к целевому шарду на основе ключа `channel_id`.
3. **Redis:** Балансировка запросов на чтение распределяется между Master и Slave узлами на уровне клиентской библиотеки (Client-side balancing).

### Схема резервного копирования

* **PostgreSQL (Метаданные):** Ежедневный полный бэкап (Full Dump) со Slave-сервера. Непрерывное архивирование WAL-логов (Write Ahead Log) в удаленное объектное хранилище (S3) для возможности восстановления на любой момент времени (Point-in-Time Recovery).
* **MongoDB (История сообщений):** Создание Snapshot-ов на уровне файловой системы (LVM/EBS) с Replica-узлов в периоды минимальной нагрузки. Бэкапы выполняются распределенно на каждом шарде.
* **Redis (Сессии):** Включен механизм AOF (Append Only File) с синхронизацией (`fsync`) каждую секунду. Раз в час создается RDB-снапшот для быстрого прогрева кэша после перезагрузки. Данные `presences` не резервируются в связи с их транзитным характером.

## 7. Алгоритмы

Для обеспечения отказоустойчивости и решения специфических проблем веерной рассылки (Fan-out) в высоконагруженных сообществах применяются специализированные инфраструктурные алгоритмы.

| Алгоритм | Область применения | Детальное описание |
| :--- | :--- | :--- |
| **Алгоритм "Пассивных сессий" (Guild Tracker / Smart Fan-out)** | Маршрутизация сообщений и статусов (Gateway / Presence) для серверов с аномально большим количеством участников (до 1 млн). | **Проблема:** При классической модели Pub/Sub публикация одного сообщения в канале сервера-миллионника вызывает лавинообразную генерацию 1 000 000 событий (O(N)), что приводит к исчерпанию пропускной способности сети и CPU.<br><br>**Решение:** Внедрение концепции состояний подписки. Когда клиент подключается к WebSocket-шлюзу, он не подписывается на все события всех своих серверов. <br>1. *Активная сессия:* Пользователь держит сервер/канал открытым на экране. Он получает полный поток данных (сообщения, тайпинг, статусы онлайна).<br>2. *Пассивная сессия:* Сервер находится в фоновом списке. Для него алгоритм фильтрует 99% трафика на стороне бэкенда. Брокер сообщений отправляет такому клиенту только критические мутации: упоминания (`@mentions`) и инкремент счетчика непрочитанных сообщений (badge update).<br><br>**Итог:** Снижение эффективного Fan-out с десятков тысяч до ~20-50 активных слушателей на канал, что экономит до 90% исходящего трафика (экономия ~1.5 ПБ/сутки согласно расчетам). |
| **Snowflake ID (Генерация распределенных ключей)** | Генерация первичных ключей (Primary Keys) для высоконагруженных сущностей без использования БД (`messages`, `users`, `channels`). | **Проблема:** Использование автоинкремента (`SERIAL` / `AUTO_INCREMENT`) в БД создает единую точку отказа и блокировки при записи (Write Bottleneck при 46 000 RPS). Использование UUID решает проблему генерации, но занимает 128 бит и не имеет строгой хронологической сортировки, что приводит к фрагментации B-Tree индексов и деградации производительности БД (Page Faults).<br><br>**Решение:** Алгоритм децентрализованной генерации 64-битных целых чисел (`bigint`) на стороне бэкенд-узлов (Application Layer). Структура ключа:<br>- *41 бит:* Timestamp (время в миллисекундах с заданной эпохи). Гарантирует сортируемость по времени создания напрямую по ключу.<br>- *10 бит:* Machine ID (уникальный идентификатор процесса/воркера). Предотвращает коллизии между разными серверами.<br>- *12 бит:* Sequence (локальный счетчик от 0 до 4095). Позволяет одному узлу генерировать до 4096 уникальных ID в миллисекунду.<br><br>**Итог:** Возможность генерации миллионов ключей в секунду без походов в БД с сохранением идеальной локальности для B-Tree индексов. |
